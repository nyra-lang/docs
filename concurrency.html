Concurrency

spawn with capture lowering, Send/Sync checks, and pthread-backed channels.

On this page
  • spawn
  • parallel for
  • progress for
  • Captures & ownership
  • Runtime API
  • Channels

spawn

spawn {
    print("runs concurrently")
}

The compiler lowers each spawn to a thread entry function plus a call to spawn_capture. Captured outer variables are packed into a struct, copied to the heap, and passed to a new thread (pthread on Unix; CreateThread on Windows).

parallel for

Use parallel for when iterations are independent. Nyra schedules worker threads; the OS maps them to CPU cores (P-cores, E-cores, etc.).

Three levels of control

// 1. Auto (recommended)
parallel for i in 0..1000 {
    work(i)
}

// 2. Cap workers (up to N; fewer if iteration count is small)
parallel(max_threads = 4) for i in 0..1000 {
    work(i)
}

// 3. Exact worker count
parallel(threads = 4) for i in 0..1000 {
    work(i)
}

More options

parallel(cpu = 80%) for i in 0..n { work(i) }          // 80% of logical CPUs
parallel(threads = cpu_count() - 1) for i in 0..n { } // leave one for the OS

parallel(mode = auto) for i in 0..n { }            // default
parallel(mode = balanced) for i in 0..n { }         // performance + responsiveness
parallel(mode = max_performance) for i in 0..n { }  // use all logical CPUs
parallel(mode = background) for i in 0..n { }      // lighter load

Prefer threads / max_threads over cores — Nyra creates workers; the OS schedules them. cores = N is accepted as an alias for max_threads with a parser note.

Built-in cpu_count() returns logical CPU count (prelude). See also stdlib/os/cpu.ny.

RuleWhy
No breakIterations run concurrently; there is no ordered early exit
No mutation of outer variablesWould be a data race without explicit locks
Captures must be SendSame thread-safety rules as spawn
Range or fixed array / string / vec_strIterable length must be known before workers start

On wasm32-wasi, parallel for runs sequentially (no pthreads).

progress for

Sequential loops with a built-in progress bar and status line — no manual print or formatting.

progress(label = "parser tests") for item in tests {
    run(item)
}

progress for i in 0..100 {
    step(i)
}

Each iteration prints:

[#####-------] 43%
Running parser tests...

Without label, the status line defaults to Running <iterable>... (e.g. Running tests...) or Processing... for ranges. Implemented in stdlib/rt/rt_progress.c; cannot combine with parallel for.

Captures & ownership

fn main() {
    let n = 42
    spawn {
        print(n)   // n captured by value (Copy)
    }
}
fn main() -> void {
    let n: i32 = 42
    spawn {
        print(n)   // n captured by value (Copy)
    }
}
Capture kindCompiler rule
Copy (i32, bool)Copied into capture struct
Move (string, structs)Must be Send; marked moved in parent scope
Active & / &mutCompile error

Channels

Implemented in stdlib/rt/rt_channel.c (and typed wrappers in stdlib/sync/channel.ny) using pthread mutex and condition variables.

Stdlib wrappers: stdlib → Sync, context & async channels (Channel_i32, Mutex, Context_with_timeout).

Runtime symbolRole
channel_new()Create channel; returns handle
channel_send(ch, value)Send i32
channel_recv(ch)Blocking receive
channel_free(ch)Release channel memory

Declare extern in your module or see Standard library → Channels.
